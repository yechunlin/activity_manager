# 爱秀活动管理 Skill

## 功能
- 管理"爱秀活动"
- 自动解析用户自然语言中的活动类型，字段
- 多轮对话支持补充活动参数

---

## 触发条件
用户表达意图：
- 创建爱秀活动，编辑爱秀活动，修改爱秀活动
- 创建报名活动，编辑报名活动，修改报名活动
- 发布报名表，编辑/修改报名表

---

## 执行规则（必须严格遵守）
- 必须按照 *_step_1 → *_step_2 → *_step_3 → *_step_4 → ... 顺序执行
- 每一步必须有明确输出结果，禁止跳步
- JSON输出必须合法、完整

---

## 识别流程模式
- 若用户意图为“创建” → 进入创建流程 create_step_1
- 若用户意图为“编辑|修改” → 进入编辑流程 edit_step_1

---

## 创建工作流程

### create_step_1：解析用户描述（活动基础信息）

#### 目标

从用户输入中提取并生成以下信息：
- 活动类型 act_type
- 活动标题 title
- 活动介绍 brief

---

#### 执行规则

判断活动类型 act_type：

根据用户描述进行语义匹配，目前仅支持：
```json
{"designh5":"报名"}
```
- 若用户明确或隐含表达为报名活动 → act_type = designh5
- 若无法判断 → 必须询问用户： “目前只支持报名活动，请确认是否创建报名活动”

---

生成活动标题 title：

- 根据用户描述生成
- 要求：
  - 简洁明确（10~20字）
  - 能体现活动主题
  - 不要包含多余修饰词

---

生成活动介绍 brief：

- 根据用户描述生成
- 要求：
  - 字数：100~150字
  - 内容完整，围绕活动目的、参与方式、亮点
  - 语言自然、清晰，不要堆砌

---

#### 注意事项

- 不要编造用户未提及的具体信息（如时间、地点等）
- 可以做合理泛化，但必须符合“报名活动”语境
- 输出必须包含：act_type、title、brief

---

#### 输出格式（必须严格遵守）

```json
{
  "act_type": "designh5",
  "title": "生成的活动标题",
  "brief": "生成的活动介绍"
}
```
---

### create_step_2：生成报名字段（fields）

#### 目标

从用户描述中提取报名表单字段，并生成结构化的 fields JSON。

---

#### 执行规则

从用户输入中提取字段信息：

- 每一个用户提到的“收集信息项”都必须转为一个字段
- 示例：
  - “姓名” → 字段
  - “手机号” → 字段
  - “上传照片” → 字段

---

生成 fields JSON：

标准格式如下：
```json
[
  {
    "label": "字段名称",
    "type": "字段类型",
    "isRequire": true,
    "placeholder": "提示文案"
  }
]
```

---

#### 字段类型映射规则（必须遵守）

##### 1. 文本输入类

适用于：

- 姓名 / 手机号 / 学号 / 公司 / 地址 等
```json
{"type": "Text"}
```

---

##### 2. 上传类（MyUpload）

当字段包含“上传/提交文件”语义时使用：
```json
{"type": "MyUpload"}
```

必须补充：
```json
{"fileType": "类型"}
```

映射规则：
- 图片 / 照片 / 头像 → picture
- 视频 / 录像 → video
- 音频 / 录音 → audio
- 文档 / 表格 / 证明 → doc

---

##### 3. 选择类（MySelect / MyRadio / MyCheckbox）

当字段包含“选择/选项”语义时使用：
```json
{"type": "MySelect"}
```

必须包含：
```json
{"options": ["选项1", "选项2"]}
```

---

#### 特殊处理（必须执行）

##### 情况1：无法推断 options

必须询问用户：
> “该字段有哪些可选项？”

在用户补充前：
- 不得生成该字段的完整结构
- 可以先记录字段名称，等待补充

---

##### 情况2：用户未提供任何字段
fields = []

---

#### 注意事项

- 所有字段默认：
```json
{"isRequire": true}
```

- placeholder 规则：
  - 文本类 → “请输入XXX”
  - 上传类 → “请上传XXX”
  - 选择类 → “请选择XXX”

- 字段名称 label 必须与用户语义一致，不要随意改写
- 不要生成用户未提及的字段

---

#### 输出格式（必须严格遵守）
```json
[
  {
    "label": "姓名",
    "type": "Text",
    "isRequire": true,
    "placeholder": "请输入姓名"
  }
]
```


### create_step_3：补充报名字段（多轮询问，必须执行）

#### 目标

确保 fields 完整、可用，必要时通过询问用户补充缺失字段。

---

#### 执行规则（必须严格遵守）

本步骤必须执行一次询问，不可跳过。

---

#### 判断逻辑

##### 情况1：fields 为空

向用户提问：
> “请告诉我需要收集哪些报名信息（如：姓名、手机号等）”

---

##### 情况2：fields 不为空

向用户提问：
> “是否还需要补充其他报名信息？”

---

#### 用户响应处理

根据用户回答执行：

---

##### 用户补充了字段

- 将新字段加入 fields
- 新字段必须重新走 create_step_2 的字段生成规则（类型、placeholder 等）

---

##### 用户明确表示“没有 / 不需要 / 就这些”

- 不再继续询问
- 进入下一步骤

---

##### 用户回答模糊（如：“随便”“你看着办”）

按以下默认逻辑处理：

```json
[
  {"label":"姓名","type":"Text","isRequire":true,"placeholder":"请输入姓名"},
  {"label":"手机号","type":"Text","isRequire":true,"placeholder":"请输入手机号"}
]
```

---

##### 当前 fields 仍为空（极端情况）

必须使用默认字段：

```json
[
  {"label":"姓名","type":"Text","isRequire":true,"placeholder":"请输入姓名"},
  {"label":"手机号","type":"Text","isRequire":true,"placeholder":"请输入手机号"}
]
```

---

#### 注意事项

- 只允许进行 **一轮补充询问**，禁止无限追问
- 不要重复询问相同问题
- 不要覆盖已有字段，只能追加或保留
- 所有新增字段必须符合 create_step_2 的结构规则

---

#### 输出结果要求

输出最终 fields JSON（完整结构），用于后续步骤

```json
[
  {
    "label": "姓名",
    "type": "Text",
    "isRequire": true,
    "placeholder": "请输入姓名"
  }
]
```


### create_step_4：海报处理（必须执行）

#### 目标

获取或生成活动海报信息，输出标准结构 post_img。

---

#### 执行规则

1必须先向用户发起询问（不可跳过）：
> “是否有现成的海报图？如果有请提供链接”

---

根据用户回复分两种情况处理：

---

#### 情况1：用户提供海报链接

判断用户回复中是否包含有效链接（URL）：
- 若包含 → 直接使用该链接

生成：

```json
{
  "url": "用户提供的海报链接",
  "size": "818*1404",
  "desc_str": ""
}
```

注意：

- 不要修改用户提供的链接
- 不要生成 desc_str

---

#### 情况2：用户没有提供海报

包括以下表达：
- “没有”
- “不用”
- “你生成”
- 未提供任何链接

执行：
1. 从用户整体描述中提取核心信息（活动主题）
2. 生成 10~50 字的简要描述 desc_str（用于生成海报）

生成：

```json
{
  "url": "",
  "size": "818*1404",
  "desc_str": "提取的活动简要描述"
}
```

---

#### 多轮交互处理（关键）

- 如果已询问但用户未明确回答：
  → 必须再次引导用户回答，不得跳过该步骤

- 如果用户先说“有”，但未提供链接：
  → 必须继续追问：
  > “请提供海报图片链接”

- 在获得明确结果（有链接 / 无海报）之前：
  → 不得进入下一步骤

---

#### 注意事项
- 当前模型不要试图生成图片，不要在线生成海报，只负责收集信息
- size 固定为："818*1404"
- desc_str 要求：
  - 10~50 字
  - 简洁描述活动主题
  - 不要堆砌修饰词
- 不要编造图片链接

---

#### 输出格式（必须严格遵守）

```json
{
  "url": "",
  "size": "818*1404",
  "desc_str": ""
}
```


### create_step_5：风格与配色生成

#### 目标

确定活动的视觉风格（预设 or 自动生成），并生成对应的配色配置。

---

#### 执行规则

必须先向用户发起选择（不可跳过）：

> “请选择一种风格，如果不合适，可以选择：自动生成配色”

必须提供“预设风格选项列表”（来自 `template/scheme_conf.json`）

---

#### 情况1：用户选择预设风格

当用户选择某个风格时：
- 记录该风格对应的 tag_id
- 不再生成 scheme JSON
- 本步骤直接结束，进入下一步

---

#### 情况2：用户选择“自动生成”

当用户表达以下语义时进入自动生成：
- “自动生成”
- “你帮我配”
- “随便配一个”
- 未选择任何具体风格

---

#### 配色生成规则（必须遵守）

根据以下信息生成：
- 活动主题（title + brief）
- 海报描述（desc_str）

---

#### 输出结构（必须严格遵守）

```json
{
  "page_config": {
    "bgColor": ""
  },
  "form": {
    "cpBorderColor": "",
    "bgColor": "",
    "titColor": "",
    "btnTextColor": "",
    "btnColor": "",
    "controlBgColor": "",
    "controlBorderColor": "",
    "controlTextColor": "",
    "controlInnerTextColor": "",
    "controlInnerPhColor": ""
  },
  "long_text": {
    "bgColor": "",
    "cpBorderColor": "",
    "color": ""
  }
}
```

---

#### 生成约束（非常重要）

- 所有颜色必须为：**合法颜色值（HEX 或 RGB）**
  - 示例：#FFFFFF / #FF5733 / rgb(255,255,255)
- 必须保证：
  - 背景色与文字颜色对比清晰（可读性强）
  - 按钮颜色突出（主色明确）
  - 整体风格统一（不要杂乱）

---

#### 多轮交互控制（必须执行）

- 如果用户未明确选择：
  → 必须再次提示用户选择，不得直接生成

- 如果用户表达模糊（如：“都行”）：
  → 视为“自动生成”

---

#### 注意事项

- 不要输出解释性文字，只输出 JSON
- 不要遗漏任何字段
- 不要生成无效颜色值（如：blue、red 等字符串）
- 不要生成过多颜色（保持简洁一致）

---

#### 输出要求总结

- 预设风格 → 只返回 tag_id
- 自动生成 → 返回完整 scheme JSON

---

### create_step_6：确认活动（不可跳过）
- 呈现之前流程生成的数据，询问是否创建活动
  - 若用户的回答里解析出语义为“是”，则创建
  - 若用户的回答里解析出语义为“否”，则中断流程，等待继续或者开启新任务


### create_step_7：创建活动（最终执行）

#### 目标

整合前面所有步骤的数据，生成最终配置文件，并调用后端接口创建活动。

---

#### 执行规则

整合所有已生成数据（必须完整）：

必须包含以下字段：

```json
{
  "act_type": "",
  "title": "",
  "brief": "",
  "post_img": {},
  "scheme": {},
  "fields": [],
  "tag_id": ""
}
```

---

#### 字段说明（必须遵守）

- act_type：来自 step_1
- title：来自 step_1
- brief：来自 step_1
- post_img：来自 step_4
- fields：来自 step_2 + step_3
- tag_id：若用户选择预设风格
- scheme：若用户选择自动生成配色

---

#### 关键约束（必须满足）

- 所有字段必须存在（不可缺失）
- 不允许为 null
- 必须符合 JSON 结构
- tag_id 和 scheme **二选一必须存在**：
  - 有 tag_id → scheme 为空对象 {}
  - 有 scheme → tag_id 为空字符串 ""

---

#### 写入临时文件

将最终 JSON 写入临时文件：
- 文件路径：由系统生成
- 内容：完整活动 JSON

---

#### 调用创建接口

调用：`scripts/create_event.py`
传入参数：
```json
{
  "temp_file_path": "临时文件路径"
}
```

---

#### 返回结果处理（必须执行）

##### 创建成功：

从返回结果中提取并展示：

- 活动标题
- 活动地址（URL）
- 平台信息
- 二维码图片（必须展示）

提示用户：
> “请使用微信扫码体验活动”

---

##### 创建失败：

直接返回：

“创建失败”

不得生成或编造任何结果

---

#### 异常处理（重要）

- 如果任一字段缺失：
  → 不得调用接口，必须回到对应步骤补全

- 如果 JSON 不合法：
  → 重新生成，不得继续执行

---

#### 注意事项

- 不要修改已有字段内容（保持前面步骤结果一致）
- 不要增加未定义字段
- 不要输出调试信息或解释
- 输出以结果为主

---

#### 输出行为总结

- 成功 → 展示活动信息 + 二维码
- 失败 → 返回“创建失败”

---

## 编辑工作流程

### edit_step_1：获取活动信息（必须）
- 提示用户：
> 请提供需要编辑修改的活动ID或活动链接
- 等待用户输入后，提取输入信息
  - 若输入为链接，从链接中提取活动ID，ID为32位字符串或者18位纯数字，无法提取时，提示用户：
  > "请提供正确的活动链接"
  - 若输入为普通字符，不满足32位字符串或者18位纯数字时，提示用户
  > "去提供正确的活动ID"
- 严格校验ID长度，32字符串或者18位纯数字，不符合的，继续提示："请提供再确认的ID或者链接"
- 正确获得获得ID后，赋值:act_id，调用`scripts/get_activity.py`，调用参数如下：
```json
{"act_id": "32cwc4wdew***********"}
```
- 获取失败时，提示：
> "活动未查询到，请确认好活动ID或链接，再次尝试"
- 获取成功时呈现活动信息：JSON格式，如下，并且要求用户确认是否正确
```json
{
  "act_id": "",
  "act_type": "",
  "title": "",
  "brief": "",
  "post": {},
  "fields": {},
  "scheme": {}
}
```
--- 

### edit_step_2：询问修改内容
- 提示：
> "请问需要修改哪些内容？如：标题、表单字段、海报、配色等"

---

### edit_step_3：解析修改意图（核心）
- 修改意图包含标题：
  - 新标题赋值给title
- 修改意图包含活动介绍(描述)
  - 新介绍(描述)赋值给brief
- 修改意图包含海报时
  - 请复用流程：create_step_4，并将结果赋值给post
- 修改意图包含配色时
  - 请复用流程：create_step_5，并将结果赋值给scheme
- 修改意图包含表单字段时
  - 若有新增，请提取分析新增字段，学习流程：create_step_2，生成字段格式
  - 若有修改，请提取分析字段，修改字段元素
  - 若有删除，请取分析字段，直接删除即可
  - 请注意，整体表单字段的格式要遵循create_step_2的要求，保持一致
- 每一步修改后都必须询问：
> "请问还需要修改其他数据吗"
- 找到判断到用户给出“否定”回到时，才结束本流程，否则一直轮询
- 结束后，将需要修改的信息呈现给用户

--- 

### edit_step_4：确认修改活动（必须）
- 提示：
> "是否确认更新活动？"

---

### edit_step_5: 提交更新
- 将要更新的数据写入临时文件，并返回临时文件地址：
- 调用`scripts/update_event.py`，传入参数{"temp_file_path":f"{临时文件地址}"}
- 修改失败时，直接显示修改失败
- 修改成功时返回：
  - 活动标题：title
  - 活动链接：url
